# Writing Style

Use this default style until the user provides writing samples or feedback.

## Audience

Write primarily for:

- The user learning or deepening a topic
- Beginner learners reading a technical blog
- The user's later review and interview preparation

Assume the reader is willing to learn but may not know the internal mechanism yet.

## Tone

- Clear Chinese technical blog style
- Natural, systematic, and practical
- Not overly academic
- Not shallow or slogan-like
- Avoid excessive jargon before the concept is explained

## Explanation pattern

Prefer this explanation rhythm:

1. Start with why the topic matters.
2. Give an intuitive explanation first.
3. Define the key terms.
4. Explain the mechanism step by step.
5. Use examples, diagrams, and comparisons for hard points.
6. Summarize what the reader should remember.

## Depth

Default to “beginner-friendly but not superficial.”

This means:

- First make the concept understandable.
- Then explain the important mechanism and boundaries.
- Add deeper implementation details only when the task asks for them or the topic requires them.

## Avoid

Avoid:

- Only listing terms without explaining relationships.
- Turning every article into a rigid interview Q&A.
- Adding sections that do not fit the current topic.
- Adding diagrams that are decorative rather than explanatory.
- Making unsupported claims without checking sources when correctness matters.

## Good habits

- Use “先看结论 / 先建立直觉 / 再看细节” style when helpful.
- Use tables when comparing concepts or solutions.
- Use diagrams for mechanisms, flows, lifecycles, and relationships.
- Use short summaries after complex sections when useful.
- End with a final summary that helps review the article later.

## How to incorporate user feedback

When the user critiques an article, generalize the feedback into a reusable style rule.

Examples:

- If the user says “这篇太像八股了,” reduce Q&A-style sections and use more reasoning-driven explanation.
- If the user says “图没解释清楚,” make each diagram include a purpose, caption, and textual walkthrough.
- If the user says “太浅了,” add mechanism details, edge cases, and comparisons without losing beginner-friendly structure.
- If the user says “结构太死,” choose fewer template sections and organize around the topic’s natural learning path.
